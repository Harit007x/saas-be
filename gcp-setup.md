# GCP Cloud Run Setup Guide

This guide contains the instructions to prepare your Google Cloud project (`nezmuzgcp`) for automatic deployments from your GitHub repository `Harit007x/saas-be`.

## Step 1: Execute Setup Script in Cloud Shell

1. Open the [Google Cloud Console](https://console.cloud.google.com/).
2. Select your project: **`nezmuzgcp`**.
3. Open **Cloud Shell** (the terminal icon `>_` in the top right corner).
4. **Copy and paste the following entire block** into the Cloud Shell and hit **Enter**:

```bash
# Set variables
export PROJECT_ID="nezmuzgcp"
export REGION="asia-south1"
export REPO_NAME="saas-repo"
export SERVICE_NAME="saas-be"
export GITHUB_REPO="Harit007x/saas-be"
export SERVICE_ACCOUNT="github-actions-deploy"
export SA_EMAIL="$SERVICE_ACCOUNT@$PROJECT_ID.iam.gserviceaccount.com"

# Ensure gcloud uses your project
gcloud config set project $PROJECT_ID

# 1. Enable required APIs
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  iamcredentials.googleapis.com \
  secretmanager.googleapis.com \
  cloudbuild.googleapis.com

# 2. Create Artifact Registry (Docker repo) in asia-south1
gcloud artifacts repositories create $REPO_NAME \
  --repository-format=docker \
  --location=$REGION \
  --description="Docker repository for SaaS backend"

# 3. Create a dedicated Service Account for GitHub Actions
gcloud iam service-accounts create $SERVICE_ACCOUNT \
  --display-name="GitHub Actions Deploy Service Account"

# 4. Grant necessary permissions (Roles) to the Service Account
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/artifactregistry.writer"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/run.developer"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/iam.serviceAccountUser"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:$SA_EMAIL" \
  --role="roles/secretmanager.secretAccessor"

# 5. Set up Workload Identity Federation (WIF)
gcloud iam workload-identity-pools create "github-pool" \
  --project=$PROJECT_ID \
  --location="global" \
  --display-name="GitHub Actions Pool"

gcloud iam workload-identity-pools providers create-oidc "github-provider" \
  --project=$PROJECT_ID \
  --location="global" \
  --workload-identity-pool="github-pool" \
  --display-name="GitHub provider" \
  --attribute-mapping="google.subject=assertion.sub,attribute.actor=assertion.actor,attribute.repository=assertion.repository" \
  --issuer-uri="https://token.actions.githubusercontent.com"

# Allow the GitHub repo to authenticate as the Service Account
export PROJECT_NUMBER=$(gcloud projects describe $PROJECT_ID --format="value(projectNumber)")

gcloud iam service-accounts add-iam-policy-binding $SA_EMAIL \
  --project=$PROJECT_ID \
  --role="roles/iam.workloadIdentityUser" \
  --member="principalSet://iam.googleapis.com/projects/$PROJECT_NUMBER/locations/global/workloadIdentityPools/github-pool/attribute.repository/$GITHUB_REPO"

# Print the sensitive details needed for GitHub Secrets
echo "=========================================================="
echo "✅ SETUP COMPLETE. COPY THE FOLLOWING TWO LINES TO GITHUB:"
echo "=========================================================="
echo "WIF_PROVIDER: projects/$PROJECT_NUMBER/locations/global/workloadIdentityPools/github-pool/providers/github-provider"
echo "SA_EMAIL: $SA_EMAIL"
echo "=========================================================="
```

## Step 2: Add Secrets to GitHub

1. Go to your GitHub repository `Harit007x/saas-be`.
2. Click on **Settings** -> **Secrets and variables** -> **Actions**.
3. Under "Repository secrets", click **New repository secret**.
4. Create the first secret:
   - **Name**: `WIF_PROVIDER`
   - **Secret**: (Paste the `WIF_PROVIDER` value generated by the script in Step 1)
5. Create the second secret:
   - **Name**: `SA_EMAIL`
   - **Secret**: (Paste the `SA_EMAIL` value generated by the script in Step 1)

## Step 3: Add your Database URL to GCP Secret Manager

Because we shouldn't save database passwords in command-line histories, you must set up your MongoDB Atlas `DATABASE_URL` via the Google Cloud Console:
1. Go to **Security > Secret Manager** in the [GCP Console](https://console.cloud.google.com/security/secret-manager).
2. Click **+ CREATE SECRET**.
3. Set **Name** to: `DATABASE_URL`
4. Under **Secret value**, explicitly paste your MongoDB Atlas URL (e.g. `mongodb+srv://<username>:<password>@cluster0...`).
5. Click **Create secret**.

*(Note: The setup script in Step 1 already gave our GitHub Actions service account access to read this secret on deployment, which will seamlessly inject it into your Cloud Run environment.)*

## Step 4: Finalize Domain Mapping

Once we push the `.github/workflows/deploy.yml` and trigger a deployment, a Cloud Run Service will be created. Let's map your custom domain `api.nezmuz.harit.online`:

1. Go to **Cloud Run** in the GCP Console.
2. Click on **Manage Custom Domains**.
3. Click **ADD MAPPING**.
4. Select the `saas-be` service.
5. Choose **Verify a new domain** and enter `harit.online`, or select it if already verified.
6. Check the box to add a custom domain and enter `api.nezmuz.harit.online`.
7. Click **Continue**.
8. Cloud Run will provide you with a specific IP Address (A Record) and possibly AAAA, or a CNAME record.
9. Go to your Domain Registrar or DNS Provider (where you purchased `harit.online`) and add the specific DNS Records instructed by Cloud Run. 
*(Once updated, it may take 15 to 30 mins to propagate and for Google to automatically provision an SSL certificate for your backend)*
